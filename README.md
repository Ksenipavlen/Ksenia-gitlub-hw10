# Ksenia-gitlub-hw10
Задание 1

1.1. Восстановление данных в полном объёме за предыдущий день
Этот сценарий предполагает, что допустима потеря данных, накопленных за текущий день до момента сбоя. Основной целью здесь является наличие гарантированно рабочей копии всей базы данных на конец предыдущего дня.
Подходящие варианты:
Полное резервное копирование (Full Backup)
Это фундаментальный метод, при котором создаётся копия абсолютно всех данных. Для выполнения этого требования компания должна выполнять полное резервное копирование базы данных один раз в сутки, как правило, в ночное время, когда нагрузка на систему минимальна.
Как это работает: Каждый день (например, в 02:00 ночи) запускается процесс, который копирует все файлы базы данных в хранилище бэкапов. В результате к утру есть актуальная копия за вчерашний день.
Преимущества:
Простота восстановления: Чтобы вернуть данные, нужно просто развернуть один-единственный архив.
Целостность: Бэкап содержит всё необходимое и является самодостаточным.
Недостатки:
Длительность: Создание полной копии может занимать много времени.
Объём: Требует большого дискового пространства для хранения ежедневных копий.
Снимки системы / снапшоты (Snapshots)
Если база данных работает на современной инфраструктуре (например, в облаке или на СХД с поддержкой этой функции), можно использовать мгновенные снимки тома, на котором лежат файлы БД.
Как это работает: Администратор или скрипт создаёт "снимок" состояния диска в определённый момент времени (например, раз в 24 часа). Это не копирование всех данных, а фиксация их состояния, что происходит почти мгновенно.
Преимущества:
Скорость: Создание снимка занимает секунды.
Минимальная нагрузка: Не влияет на производительность базы данных.

1.2. Восстановление данных за час до предполагаемой поломки
Этот сценарий требует гораздо более высокой частоты создания резервных копий, так как допустимая потеря данных составляет всего один час. Здесь одного полного бэкапа в сутки недостаточно.
Подходящие варианты:
Инкрементальное или дифференциальное резервное копирование (Incremental / Differential Backups)
Эти методы используются в связке с ежедневным полным бэкапом для сокращения объёма передаваемых данных и времени создания копий.
Инкрементальный бэкап (Incremental): Копирует только те данные, которые изменились с момента последнего любого бэкапа (полного или инкрементального).
Схема: Полный бэкап (раз в день) -> Инкрементальный бэкап (каждый час).
Восстановление: Чтобы восстановить данные на 14:00, нужно развернуть полный бэкап за вчерашний день и последовательно применить все инкрементальные бэкапы с 00:00 до 14:00. Это самый быстрый способ создания ежечасных копий.
Дифференциальный бэкап (Differential): Копирует все данные, изменившиеся с момента последнего полного бэкапа.
Схема: Полный бэкап (раз в день) -> Дифференциальный бэкап (каждый час).
Восстановление: Чтобы восстановить данные на 14:00, нужно развернуть полный бэкап и только один дифференциальный бэкап, созданный в 14:00. Это самый быстрый способ восстановления, но ежечасные копии будут со временем занимать всё больше места.
Журналирование транзакций (Transaction Log Backups)
Это наиболее предпочтительный и надёжный метод для баз данных, особенно в финансовой сфере. Он позволяет достичь минимально возможного RPO.
Как это работает: Система управления базами данных (СУБД) ведёт журнал всех транзакций (изменений). Периодически (например, каждые 15-30 минут) делается резервная копия этого журнала. При восстановлении сначала накатывается полный бэкап, а затем последовательно "проигрываются" все транзакции из журналов до нужного момента времени.
Преимущества:
Точечное восстановление (Point-in-Time Recovery): Позволяет восстановить данные на любой конкретный момент времени (например, "за 5 минут до сбоя"), а не только на момент создания копии.
Минимальная потеря данных: Потеря данных ограничивается интервалом между двумя последовательными бэкапами журнала транзакций.

Задание 2
Резервное копирование (pg_dump)
Создаёт резервную копию одной базы данных mydatabase от имени пользователя myuser и сжимает её с помощью алгоритма gzip.
Команда:
pg_dump -U myuser -h localhost -F c -Z 9 -f mydatabase_backup.dump mydatabase

Восстановление (pg_restore)
Восстанавливает базу данных из файла, созданного в пользовательском формате. Перед восстановлением необходимо создать пустую базу данных.
Шаги:
Создать пустую базу данных:
CREATE DATABASE mydatabase_restored;
Выполнить восстановление:
pg_restore -U myuser -h localhost -d mydatabase_restored -v mydatabase_backup.dump

Задание 3
Для MySQL Enterprise Backup:
1. Полный бэкап (база):
mysqlbackup --user=root --password='pwd' --backup-dir=/full backup-and-apply-log
2. Инкрементный бэкап (изменения с момента полного):
mysqlbackup --user=root --password='pwd' --backup-dir=/inc1 --incremental-basedir=/full backup-and-apply-log
3. Инкрементный бэкап (изменения с момента предыдущего инкрементного):
mysqlbackup --user=root --password='pwd' --backup-dir=/inc2 --incremental-basedir=/inc1 backup-and-apply-log
Для стандартной версии MySQL (Community):
Инкрементное копирование делается через бинарные логи (--log_bin). Сначала делается полный бэкап (например, mysqldump), а затем копируются файлы логов, которые «накатываются» на него для восстановления.
